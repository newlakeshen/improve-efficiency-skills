# Codex 提示词优化

使用这个 skill，将用户的提示词草稿改写成更适合 Codex 或其他 coding agent 执行的高质量提示词。对非代码任务，也按 OpenAI prompt guidance 优化为清晰、简洁、结果导向的任务契约。

参考指南：
- https://developers.openai.com/api/docs/guides/prompt-guidance
- https://developers.openai.com/cookbook/examples/gpt-5/codex_prompting_guide

## 默认输出

默认只返回优化后的提示词。只有当用户明确要求分析、对比或解释原因时，才补充说明。

如果输入存在歧义但仍可执行，做出合理假设并继续优化提示词。只有当缺失信息会实质性改变提示词目标、安全边界、工具权限或必要输出格式时，才向用户提问澄清。

## 优化流程

1. 识别预期结果。
   - 模型应该产出什么，或完成什么？
   - 输出面向谁？
   - 什么样的结果才算“好”？

2. 移除提示词堆叠噪音。
   - 优先使用结果导向的指令，而不是冗长的流程脚本。
   - 删除冗余的角色扮演、重复约束，以及并非真正必要的逐步控制。
   - 只在真实不变量上使用 `must`、`always`、`never`、`only` 等绝对化词汇。

3. 补齐缺失的任务契约。
   只加入会改变模型行为的部分：
   - `Role`：模型的职责和工作上下文。
   - `Goal`：用户可见的最终结果。
   - `Success criteria`：最终回答前必须满足的条件。
   - `Context`：模型可以依赖的事实、来源、示例或材料。
   - `Constraints`：安全、隐私、业务、证据、副作用或范围限制。
   - `Output`：必要的格式、长度、语气、结构和语言。
   - `Stop rules`：何时回答、重试、提问、拒答或停止。

4. 为 Codex 补齐执行契约。
   代码任务通常需要明确这些内容：
   - `Repo context`：目标仓库、相关路径、入口、现有约定或需要先读取的文件。
   - `Autonomy`：是否应直接实现、何时提问、遇到阻塞时如何推进。
   - `Tool rules`：读取、搜索、编辑、联网、运行命令、使用 subagent 或 browser 的边界。
   - `Change scope`：允许修改哪些文件，哪些文件、行为或接口不能动。
   - `Validation`：必须运行的测试、构建、lint、截图、回归检查或人工验收标准。
   - `Final report`：最终应说明改了什么、验证结果、未完成事项和风险。

5. 调整推理和工具使用行为。
   - 对简单任务，保持提示词简短，避免加入沉重的工作流规则。
   - 对工具密集或耗时较长的任务，要求在使用工具前先给出简短的用户可见说明。
   - 当任务需要研究或引用时，定义检索预算。
   - 当任务可以测试、渲染、计算或验证时，加入验证检查。
   - 对高影响或不可逆操作，要求执行前先做预检查。
   - 对代码修改任务，要求先读现有实现，再按最小必要范围编辑，最后验证。

6. 保留真正重要的信息。
   - 保留用户的任务意图、领域、受众、语言和硬性要求。
   - 不要添加没有依据的产品声明、指标、政策或能力描述。
   - 不要为了显得完整而让提示词变长；只有当额外具体性会改善行为时才增加内容。

## 建议提示词结构

对非简单 Codex 任务，使用这个紧凑结构：

```text
Role: You are Codex working in [repo/workspace]. [用 1-2 句话定义职责]

# Goal
[Codex 应该完成的具体结果]

# Success criteria
[正确最终回答需要满足的可观察要求]

# Context
[仓库、路径、需求、约定、示例、来源或用户提供材料]

# Constraints
[范围、证据、安全、隐私、权限、不可改动项和副作用限制]

# Workflow
[需要探索、实现、验证、汇报的关键步骤；只保留会改变执行行为的步骤]

# Validation
[必须运行或说明无法运行的测试、构建、lint、截图或检查]

# Final response
[格式、长度、语气、语言、必须汇报的信息]
```

省略空章节。对小型提示词，把结果压缩成一个短段落，而不是强行套用完整结构。对纯问答或写作任务，可以保留通用结构：`Goal`、`Context`、`Constraints`、`Output`。

## Codex 特别注意

- 不要要求 Codex 暴露隐藏推理；需要理由时，要求给出简短、可审计的决策依据。
- 不要把“先给计划”作为默认要求。只有高风险、范围不清或用户明确要求时，才让 Codex 先停下来确认。
- 对执行类任务，提示词应鼓励 Codex 持续推进到实现和验证完成，而不是只给建议。
- 对有副作用的操作，明确哪些操作可直接执行，哪些需要用户确认，例如发布、删除数据、迁移生产库、提交或 push。
- 对前端任务，要求可视化验证；对数据、交易、财务、医疗、法律等高影响任务，要求来源、边界和失败处理。

## 质量检查清单

返回优化后的提示词前，确认：

- 提示词是结果导向且简洁的。
- 成功标准足够明确，可以判断任务是否完成。
- 约束是真实约束，而不是装饰性文字。
- 输出契约清晰。
- 在相关场景中，已经定义缺少证据或需要澄清时的行为。
- 只有在任务需要时，才加入验证或引用规则。
- 对代码任务，已经定义探索、编辑范围、验证和最终汇报要求。
- 没有要求模型输出隐藏推理或执行不必要的流程仪式。
- 改写后的提示词没有改变用户原意。
