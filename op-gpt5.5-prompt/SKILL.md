# GPT-5.5 提示词优化

使用这个 skill，将用户的提示词草稿改写成更适合 GPT-5.5 的高质量提示词。

参考指南：https://developers.openai.com/api/docs/guides/prompt-guidance

## 默认输出

默认只返回优化后的提示词。只有当用户明确要求分析、对比或解释原因时，才补充说明。

如果输入存在歧义但仍可执行，做出合理假设并继续优化提示词。只有当缺失信息会实质性改变提示词目标、安全边界、工具权限或必要输出格式时，才向用户提问澄清。

## 优化流程

1. 识别预期结果。
   - 模型应该产出什么，或完成什么？
   - 输出面向谁？
   - 什么样的结果才算“好”？

2. 移除提示词堆叠噪音。
   - 优先使用结果导向的指令，而不是冗长的流程脚本。
   - 删除冗余的角色扮演、重复约束，以及并非真正必要的逐步控制。
   - 只在真实不变量上使用 `must`、`always`、`never`、`only` 等绝对化词汇。

3. 补齐缺失的任务契约。
   只加入会改变模型行为的部分：
   - `Role`：模型的职责和工作上下文。
   - `Goal`：用户可见的最终结果。
   - `Success criteria`：最终回答前必须满足的条件。
   - `Context`：模型可以依赖的事实、来源、示例或材料。
   - `Constraints`：安全、隐私、业务、证据、副作用或范围限制。
   - `Output`：必要的格式、长度、语气、结构和语言。
   - `Stop rules`：何时回答、重试、提问、拒答或停止。

4. 调整推理和工具使用行为。
   - 对简单任务，保持提示词简短，避免加入沉重的工作流规则。
   - 对工具密集或耗时较长的任务，要求在使用工具前先给出简短的用户可见说明。
   - 当任务需要研究或引用时，定义检索预算。
   - 当任务可以测试、渲染、计算或验证时，加入验证检查。
   - 对高影响或不可逆操作，要求执行前先做预检查。

5. 保留真正重要的信息。
   - 保留用户的任务意图、领域、受众、语言和硬性要求。
   - 不要添加没有依据的产品声明、指标、政策或能力描述。
   - 不要为了显得完整而让提示词变长；只有当额外具体性会改善行为时才增加内容。

## 建议提示词结构

对非简单提示词，使用这个紧凑结构：

```text
Role: [用 1-2 句话定义模型的功能、上下文和职责]

# Goal
[模型应该达成的具体结果]

# Success criteria
[正确最终回答需要满足的可观察要求]

# Context
[可使用的事实、材料、示例或来源]

# Constraints
[范围、证据、安全、隐私、权限和副作用限制]

# Output
[格式、长度、语气、语言和必要章节]

# Stop rules
[何时回答、询问缺失信息、重试、降级处理或停止]
```

省略空章节。对小型提示词，把结果压缩成一个短段落，而不是强行套用完整结构。

## 质量检查清单

返回优化后的提示词前，确认：

- 提示词是结果导向且简洁的。
- 成功标准足够明确，可以判断任务是否完成。
- 约束是真实约束，而不是装饰性文字。
- 输出契约清晰。
- 在相关场景中，已经定义缺少证据或需要澄清时的行为。
- 只有在任务需要时，才加入验证或引用规则。
- 改写后的提示词没有改变用户原意。
